# Deep Obligations — How-to-Check 详细文稿

> 本文档整理 C1、D1、D2、E1、E2 五条 deep 义务的 how_to_check 完整内容，
> 供面试脱稿复述和对抗式评审准备使用。
> 每条义务附有：判断逻辑分步说明、实习生测试自检、反向测试要点。

## C1 — 非面对面开户的额外保障措施

### 法条依据
MLR 2017 Reg 33(6)(b)(iii)（非面对面是 EDD 触发因素）、Reg 28(19)（电子身份验证标准）、JMLSG Part I Ch.5 para 5.3.5

### 触发条件判断逻辑

判断这条义务是否适用，核心是回答一个问题：**整个开户流程里，有没有任何一个环节是客户与银行工作人员之间实时人际互动的？**

具体来说：
- 如果流程中存在"客户到网点见面"、"客户与工作人员视频通话"、"第三方见证人在场"等步骤 → 触发条件 **未激活**，本条标记为"不适用"（但混合流程仍保留部分非面对面风险，值得备注）
- 如果所有身份验证都是客户自助提交（上传证件照片、自拍、填表）或自动化系统完成（API 调用第三方验证服务），没有任何实时人工环节 → 触发条件 **激活**，进入第二步

关键区分点：Onfido 这类异步验证服务不算"实时人际互动"——客户提交材料，系统异步返回结果，中间没有人与人的交互。只有"工作人员实时看着客户完成验证"才算面对面。

### 保障措施检查逻辑

触发条件激活后，判断 BRD 是否描述了以下至少一种补偿性保障措施：

**(a) 活体检测（Liveness Detection）**
BRD 是否描述了生物识别/活体检测步骤——即不仅要求客户上传一张人脸照片，而是要求客户完成动作（眨眼、转头）或使用视频活体检测？如果 BRD 只说"上传自拍"或"拍照匹配"，没有提及活体，这不满足 (a)。

**(b) 多源交叉验证（Multi-source Verification）**
BRD 是否描述了将客户身份信息与两个或以上独立数据源进行交叉验证？例如：证件验证 + 信用局数据 + 选民登记数据。如果只用一个数据源（即使是有资质的第三方），不满足 (b)。

**(c) 数字身份服务标准引用**
BRD 是否引用了 UK Digital Identity Trust Framework（UK DITF）或 MLR 2017 Reg 28(19) 的标准要求？即不仅说"我们用 Onfido"，而是说"我们使用的数字身份服务必须满足 Reg 28(19) 的反欺诈和充分保证要求"。

**(d) 补充验证步骤**
BRD 是否描述了额外的验证步骤，例如向客户已有的银行账户发起一笔微小转账（micro-deposit verification）、基于知识的验证问题（KBA）等？

### 状态判定规则

| 情况 | 判定 |
|------|------|
| 触发条件未激活（有实时人工环节） | 不适用 |
| 触发条件激活，且 (a)(b)(c)(d) 中至少一项被描述 | 已满足 |
| 触发条件激活，且只说"上传证件照片"或"用 Onfido API"而没有任何上述保障措施 | 缺口 |

### 实习生测试

> 如果一个实习生拿到这条 how_to_check 和一份 BRD，他能判断出"已满足"还是"缺口"吗？

能。因为判断标准是具体的：先看有没有人工环节（有/没有），再看有没有四种保障措施中的任何一种（有/没有）。不需要合规经验，只需要按步骤对照。

### 反向测试要点

拿一段描述"客户到网点完成开户"的 BRD 去测——应该被判定为"不适用"，而不是误报为缺口。另一个反向测试：拿一段描述"使用 Onfido 进行证件验证 + 活体检测 + 信用局交叉验证"的 BRD 去测——应该被判定为"已满足"，不误报。

## D1 — 制裁名单筛查

### 法条依据
SAMLA 2018（Sanctions and Anti-Money Laundering Act 2018）、OFSI guidance（UK Office of Financial Sanctions Implementation）、JMLSG Part I Ch.6

### 触发条件

本项目涉及为新客户开户。制裁筛查必须在开户时执行。触发条件对任何开户项目都适用。

### 检查逻辑

**第一步：BRD 是否描述了制裁筛查步骤？**

在 BRD 描述的 onboarding 流程中，是否有一个明确的步骤是对客户进行制裁名单筛查？判断方式：
- 流程图或步骤描述中是否有"sanctions screening"、"sanctions check"、"OFAC check"、"watchlist screening"等表述
- 是否提到了具体的筛查系统或服务（如 Accuity, Dow Jones, Refinitiv World-Check）
- 是否将筛查描述为 KYC 检查的一部分

**第二步：BRD 是否指定了筛查哪些名单？**

UK 银行必须筛查的名单包括：
- (a) OFSI Consolidated List（英国制裁名单）
- (b) UN Security Council Consolidated List（联合国制裁名单）
- (c) EU Financial Sanctions List（欧盟金融制裁名单，英国脱欧后通过 retained EU law 保留）

如果 BRD 列出了至少一个上述名单 → 已满足
如果 BRD 提到"制裁筛查"但没有指定具体名单 → 不确定（标记为 unclear，建议补充）
如果 BRD 完全没有提到制裁筛查 → 缺口

**第三步：边界情况处理**

- 如果 BRD 提到一个"KYC check"并说明它包含 PEP 和 sanctions，但没有进一步拆分 → 标记为"unclear"而非"已满足"。理由：制裁筛查和 PEP 筛查是不同的义务、使用不同的数据源、有不同的处理逻辑，BRD 应该将它们分开描述
- 如果 BRD 说"制裁筛查由后台系统自动完成，不在本项目范围内" → 这需要判断：如果 BRD 明确引用了现有的筛查系统并说明本项目依赖该系统，可以标记为"已满足（依赖现有系统）"；如果只是说"不在范围内"但没有引用任何现有系统 → 缺口

### 状态判定规则

| 情况 | 判定 |
|------|------|
| BRD 描述了制裁筛查步骤，且指定了至少一个名单 | 已满足 |
| BRD 描述了制裁筛查步骤，但未指定名单 | 不确定 |
| BRD 提到"KYC check"含制裁筛查但未拆分 | 不确定 |
| BRD 引用现有筛查系统并说明依赖关系 | 已满足（依赖现有系统） |
| BRD 完全未提及制裁筛查 | 缺口 |

### 实习生测试

> 实习生拿到这份 how_to_check 和一份 BRD，能否判断？

能。核心判断是"BRD 里有没有描述制裁筛查步骤"以及"有没有指定名单"。这两件事都是文本层面可以确认的，不需要合规判断力。

### 反向测试要点

拿一段完全不涉及开户的 BRD（比如内部系统升级）去测——应该被判定为"不适用"。拿一段只提"PEP screening"但不提制裁筛查的 BRD 去测——PEP 筛查不等于制裁筛查，应该被判定为"缺口"。

## D2 — PEP 筛查

### 法条依据
MLR 2017 Reg 35（PEP 的 EDD 要求）、FCA FG17-6（FCA 关于 PEP 处理的最终指引）、JMLSG Part I Ch.5 para 5.5

### 触发条件

本项目涉及为新客户开户。PEP 筛查必须在开户时执行。触发条件对任何开户项目都适用。

### 检查逻辑

**第一步：BRD 是否描述了 PEP 筛查步骤？**

在 BRD 描述的 onboarding 流程中，是否有一个明确的步骤是对客户进行 PEP 筛查？判断方式：
- 流程图或步骤描述中是否有"PEP screening"、"PEP check"、"politically exposed person check"等表述
- 是否提到了具体的 PEP 数据服务（如 World-Check, Dow Jones Risk & Compliance）
- 是否将 PEP 筛查描述为 KYC 检查的一部分

**第二步：BRD 描述的筛查范围是否覆盖了三类对象？**

PEP 筛查不仅是检查客户本人是否是 PEP，还必须覆盖：
- (a) 客户本人（是否是 PEP？）
- (b) PEP 的家庭成员（配偶、子女、父母）
- (c) PEP 的已知密切关联人（known close associates）

如果 BRD 只说"PEP check"但没有说明覆盖范围 → 可以标记为"已满足"（方法论细节可能在操作文档中），但如果要达到 deep 级别的严谨度，应该备注"建议明确筛查覆盖家庭成员和密切关联人"

**第三步：BRD 是否区分了国内 PEP 和外国 PEP？**

FCA FG17-6 明确：国内 PEP 可以被视为比外国 PEP 更低的风险，但 EDD 仍然必须适用——只是强度可以降低。如果 BRD：
- 对所有 PEP 统一适用 EDD → 可接受（保守做法），已满足
- 对国内 PEP 完全免除 EDD → 缺口（国内 PEP 仍需 EDD）
- 未区分但对所有 PEP 适用 EDD → 已满足

**第四步：与 C2 的依赖关系**

D2 是 C2 的前置条件——如果 D2（PEP 筛查）没有描述，C2（PEP 触发 EDD）也无法被满足，因为触发条件无法被检测到。在检查 D2 时应备注这条依赖关系。

### 状态判定规则

| 情况 | 判定 |
|------|------|
| BRD 描述了 PEP 筛查步骤 | 已满足 |
| BRD 描述了 PEP 筛查且覆盖家庭成员和密切关联人 | 已满足（最佳实践） |
| BRD 提到"KYC check"含 PEP 但未拆分 | 不确定 |
| BRD 未提及 PEP 筛查 | 缺口 |
| BRD 对国内 PEP 免除 EDD | 缺口 |

### 实习生测试

> 实习生拿到这份 how_to_check 和一份 BRD，能否判断？

能。核心判断是"BRD 里有没有描述 PEP 筛查步骤"。这是文本层面可以直接确认的。对国内/外国 PEP 的区分是进阶判断，但基础判断（有没有 PEP 筛查）不需要经验。

### 反向测试要点

拿一段只提"sanctions screening"但不提 PEP 筛查的 BRD 去测——制裁筛查不等于 PEP 筛查，应该被判定为"缺口"。拿一段对 PEP 适用完整 EDD 的 BRD 去测——应该被判定为"已满足"，不误报。

## E1 — 资金来源信息收集

### 法条依据
JMLSG Part I Ch.5 para 5.3.4、MLR 2017 Reg 28(2)(c)（获取业务关系目的和性质的信息）

### 触发条件

本项目涉及开立新账户。资金来源（Source of Funds, SOF）信息应作为所有客户 CDD 的一部分获取，对高风险客户需要进一步验证。

注意区分 SOF 和 SOW：
- **SOF（Source of Funds）**= 这笔钱从哪里来？（例如：工资、储蓄、转账、遗产）
- **SOW（Source of Wealth）**= 客户的全部财富是如何积累的？（这是 E2 的义务）

### 检查逻辑

**第一步：BRD 是否描述了 SOF 信息的收集步骤？**

在 BRD 描述的 onboarding 流程中，是否有一步是收集客户的资金来源信息？判断方式：
- 是否有一个表单字段让客户声明资金来源
- 是否描述了收集就业状态和收入范围
- 是否描述了对初始存款来源的询问

**第二步：SOF 收集的形式是什么？**

SOF 收集可以采取以下形式之一：
- (a) 声明就业状态和薪资范围（如"employed, £40k-£60k"）
- (b) 声明初始存款来源（如"salary"、"savings"、"transfer from another account"）
- (c) 对大额初始存款的文件要求（如"deposits over £X require source documentation"）

**第三步：SOF 收集的范围是否完整？**

- 如果 BRD 对所有客户都收集 SOF → 已满足
- 如果 BRD 只在大额存款时收集 SOF，没有一般性收集机制 → 部分缺口
- 如果 BRD 完全没有 SOF 收集步骤 → 缺口

**第四步：与 A6 的关联**

SOF 收集与 A6（开户目的和预期交易活动）密切相关。A6 问的是"你为什么开户、预期怎么用"，E1 问的是"开户的钱从哪来"。两者都是 Reg 28(2)(c) 的要求，但关注点不同。在检查 E1 时应确认 A6 是否也已满足。

### 状态判定规则

| 情况 | 判定 |
|------|------|
| BRD 对所有客户收集 SOF 信息（至少一种形式） | 已满足 |
| BRD 只在大额存款时收集 SOF | 部分缺口 |
| BRD 完全没有 SOF 收集步骤 | 缺口 |

### 实习生测试

> 实习生拿到这份 how_to_check 和一份 BRD，能否判断？

能。核心判断是"BRD 里有没有让客户声明资金来源的步骤"。这是流程描述层面可以确认的，不需要金融知识。SOF 和 SOW 的区分可能需要额外解释，但 how_to_check 里已经写清楚了。

### 反向测试要点

拿一段收集了"就业状态"但只用于营销、不用于 CDD 的 BRD 去测——应被判定为"部分缺口"而非"已满足"。拿一段完整收集 SOF 且用于风险评估的 BRD 去测——应被判定为"已满足"，不误报。

## E2 — EDD 场景的资金来源与财富来源验证

### 法条依据
MLR 2017 Reg 33(3A)(c)（EDD 必须验证 SOW 和 SOF）、JMLSG Part I Ch.5 para 5.5.5

### 触发条件

客户触发了 EDD（增强尽职调查）。EDD 的触发条件包括：
- 客户是 PEP（C2）
- 客户来自高风险第三国（C3）
- 交易复杂/异常大/无明确目的（C4）
- 非面对面开户无补偿保障（C1）
- 发现虚假身份文件（C5）
- 风险评估评为高风险（C6）

只要上述任何一项触发，E2 即适用。

### 检查逻辑

**第一步：BRD 是否描述了 SOW 验证步骤？**

关键区分：SOW 验证 ≠ SOF 收集。
- SOF 收集（E1）= 问客户"这笔钱从哪来"，客户口头声明即可
- SOW 验证（E2）= 要求客户提供**证据**证明其财富来源，例如工资单、纳税申报表、房产出售合同、企业财务报表

判断 BRD 是否描述了 SOW 验证：
- 是否有一个步骤说"当客户触发 EDD 时，要求提供财富来源证明文件"
- 是否列出了可接受的证据类型（payslip, tax return, bank statement, sale agreement, business financials）
- 是否描述了验证过程（谁审核、怎么审核、审核结果如何记录）

**第二步：SOW 验证是否与 EDD 触发条件关联？**

BRD 应该描述一个逻辑：当 C1-C6 中任何一项触发时，自动激活 SOW 验证步骤。如果 BRD 描述了 SOW 验证但没有说明何时触发 → 部分缺口（有工具但没用对地方）。如果 BRD 既没有 SOW 验证步骤也没有触发逻辑 → 缺口。

**第三步：SOW 验证是否要求证据而非仅声明？**

如果 BRD 只说"问客户财富来源"但不要求文件证据 → 缺口。自声明对 EDD 场景是不够的——洗钱者可以随便说"工资"或"遗产"而不提供证明。BRD 必须描述证据要求。

**第四步：与 E3 的关系**

E3 是 E2 的细化——E3 要求 SOW/SOF 验证必须使用证据而非自声明。E2 检查的是"有没有 SOW 验证步骤"，E3 检查的是"这个步骤是否要求证据"。在检查 E2 时应确认 E3 是否也已满足。

### 状态判定规则

| 情况 | 判定 |
|------|------|
| BRD 描述了 SOW 验证步骤，与 EDD 触发条件关联，且要求证据 | 已满足 |
| BRD 描述了 SOW 验证步骤但未与 EDD 触发关联 | 部分缺口 |
| BRD 描述了 SOW 验证但不要求证据（仅声明） | 缺口 |
| BRD 完全没有 SOW 验证步骤 | 缺口 |

### 实习生测试

> 实习生拿到这份 how_to_check 和一份 BRD，能否判断？

能，但需要额外解释 SOF 和 SOW 的区别。核心判断是：(1) BRD 里有没有"EDD 时要求财富来源证明"的步骤，(2) 这个步骤是否要求文件证据。这两点都是文本层面可以确认的。关键是实习生要理解"声明"和"证据"的区别——这一点在 how_to_check 中已经写清楚了。

### 反向测试要点

拿一段对所有客户都收集 SOF 声明但没有 EDD 场景 SOW 验证的 BRD 去测——应被判定为"缺口"（E1 满足但 E2 不满足）。拿一段完整描述 EDD 触发 → SOW 证据要求 → 审核流程的 BRD 去测——应被判定为"已满足"，不误报。

## 交叉依赖关系

```
B1 (风险评估) ──触发──→ C1/C2/C3/C4/C5/C6 (EDD触发)
                              │
                              ├──→ E2 (SOW验证)
                              ├──→ C2 (PEP的EDD)
                              └──→ C6 (通用EDD包)

D2 (PEP筛查) ──触发──→ C2 (PEP触发EDD) ──触发──→ E2 (SOW验证)
D1 (制裁筛查) ──独立──→ F3 (定期重筛)

E1 (SOF收集) ──基础──→ E2 (SOW验证) ──细化──→ E3 (证据要求)
A6 (开户目的) ──关联──→ E1 (SOF收集) ──关联──→ F1 (持续监控基线)
```

理解这些依赖关系的重要性：如果 D2（PEP 筛查）是缺口，那么 C2（PEP 触发 EDD）和 E2（EDD 场景 SOW 验证）也无法被满足——因为触发条件本身无法被检测。在演示和面试中，讲清楚这些依赖关系是证明系统性思维的关键。
